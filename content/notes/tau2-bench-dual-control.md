$\tau^2$-Bench는 도구를 쓰는 대화형 에이전트(conversational agent)를 실제 운영 환경에 더 가깝게 평가하려는 벤치마크다. 핵심 아이디어는 에이전트만 데이터베이스와 도구를 쓰는 것이 아니라, 사용자 쪽도 별도의 도구와 상태를 가진 시뮬레이션 에이전트로 모델링하는 것이다. 논문은 이를 dual-control environment라고 부른다. 예를 들어 통신사 고객지원에서는 상담 에이전트가 계정 DB와 정책 도구를 조작하는 동시에, 사용자는 자신의 휴대폰 상태를 확인하거나 비행기 모드를 끄거나 SIM 카드를 다시 삽입하는 식의 작업을 수행한다.

기존 Tau-Bench 계열 평가는 항공·소매 도메인에서 에이전트의 정책 준수와 도구 호출을 봤지만, 사용자는 주로 텍스트로 응답하는 상대였다. $\tau^2$-Bench는 여기에 Telecom 도메인을 추가해 사용자 행동과 사용자 기기 상태까지 제어한다. 그래서 에이전트가 “상태 표시줄을 확인해 달라”, “모바일 데이터를 켜 달라” 같은 실행 가능한 지시를 내리면, 사용자 시뮬레이터는 실제 mock phone 도구를 호출해 기저 상태에 맞는 결과를 돌려준다.

실무적으로 중요한 점은 평가 난도가 단순한 task success보다 더 현실적이라는 것이다. 에이전트는 고객 식별, 정책 확인, 문제 분류, 단계적 진단, 사용자에게 시킬 수 있는 행동과 에이전트 자신이 도구로 처리해야 하는 행동의 구분을 모두 맞춰야 한다. 논문은 $pass^k$와 $pass^1$ 같은 성공률 지표를 여러 모델과 운영 모드에서 비교하고, 필요한 action 수나 issue 유형이 늘어날수록 성능이 어떻게 변하는지 분석한다.

Telecom 도메인에서는 No Service, 모바일 데이터 문제, MMS 문제 같은 intent를 다루며, 각 과제는 여러 subtask로 구성된다. 표본 기준으로 114개 과제를 만들고, 전체 조합은 2285개까지 확장된다. 중요한 발견은 사용자 시뮬레이터 품질이다. 새 Telecom 도메인은 기존 Airline/Retail보다 사용자 시뮬레이터 오류율이 낮고 critical error가 보고되지 않아, 대화 실패를 더 안정적으로 에이전트 성능 문제로 해석할 수 있다.

에이전트 개발자 관점에서 이 논문은 “도구 호출을 잘하는 모델”과 “현장에서 문제를 끝까지 해결하는 에이전트” 사이의 차이를 잘 보여 준다. 실제 고객지원 자동화에서는 백엔드 API뿐 아니라 사용자가 보는 화면, 사용자가 수행할 수 있는 조작, 정책상 이관해야 하는 조건, 여러 issue가 동시에 있는 상황을 평가해야 한다. $\tau^2$-Bench는 이런 end-to-end 대화형 제어 문제를 재현하는 평가 설계로 볼 수 있다.

운영팀이 이 벤치마크를 참고한다면 최종 성공률만 보지 말고, 실패 로그에서 다음을 분리해 보는 것이 좋다. 에이전트가 잘못된 진단 순서를 택했는가, 사용자에게 불가능하거나 모호한 지시를 했는가, 정책상 사람에게 transfer해야 할 상황을 놓쳤는가, 또는 사용자 상태 변화가 DB와 일관되게 반영되지 않았는가. 이 구분이 되어야 모델 교체, policy 개선, tool schema 개선, 사용자 시뮬레이터 개선 중 무엇이 필요한지 판단할 수 있다.
